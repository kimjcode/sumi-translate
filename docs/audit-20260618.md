## 🔴 高（資安／資料外洩／會崩）

### H1. Google MT 的 API key 可能被寫進 log（紅線：key 不進 log，零容忍）
- **位置**：`src-tauri/src/providers/mt.rs:61`（key 放 URL query：`{GOOGLE_ENDPOINT}?key={api_key}`）
  + `src-tauri/src/pipeline.rs:271-273`（`ProviderError::Network(msg) => format!("network error: {msg}")`，在 `log::warn!` 裡輸出）。
- **問題**：Google 把 key 放在 query string。`mt.rs:69/75` 把 reqwest 的 `.send()`/`.text()` 錯誤
  以 `e.to_string()` 包成 `ProviderError::Network`。reqwest 的錯誤 Display **會帶上請求 URL**
  （`error sending request for url (https://…/translate/v2?key=AIza…)`）。這個字串雖然**不會顯示給使用者**
  （`user_message` 對 `Network(_)` 只回「連不上 Google」），但會被 `run_translation` 的 `log::warn!` 寫出。
  → 只要 Google 路徑發生連線層錯誤（逾時／斷網／DNS／TLS），**整把金鑰就進了 log**。
- **為什麼是問題**：直接違反 `CLAUDE.md` 紅線「任何 log 都不可含 key、外洩零容忍」。DeepL／Gemini 走 header
  （`mt.rs:145`、`llm.rs:76`），URL 不含 key，**只有 Google MT 這條中招**。
- **建議**：(a) log 端對 `Network` 不要輸出原始字串，或先做 redact（把 `key=...` 之後遮掉）；
  或 (b) 把 Google key 改放 header（`X-Goog-Api-Key: <key>`，Google v2 支援），讓 URL 永不含 key——
  這條同時根治「未來任何拿 reqwest error 字串去 log」的風險。建議兩者都做（b 為主）。

### H2. 字典 fallback（`gemini_define`）完全沒過機密過濾，且把整個原文框送 Gemini
- **位置**：`src-tauri/src/workbench.rs:242-316`（`gemini_define` 直接 `stream_generate`，**沒有 `filter::classify`**）
  + 呼叫端 `src/workbench/Workbench.tsx:196,237`（`const sentence = ta.value;` → `api.geminiDefine(word, sentence, …)`，**送的是整個原文框內容**，不是單一句子）。
- **問題**：稽核紅線明列「機密過濾在**所有**送外部 API 的路徑（含**字典 fallback**）都要生效」。
  但點到 ECDICT 查無的字時，後端**不經過濾層**就把 `word` + **整個原文框文字**送到 Gemini。
  情境：使用者在 blank Workbench（D8）或編輯框貼進含密碼／token 的設定／log，點任何生字 → 全文外送 Gemini。
  `workbench_translate`（重翻路徑）有過濾、會擋 `Secret`，但**字典路徑繞過它**。
- **為什麼是問題**：這是「機密內容不送外部」紅線在一條真實出口上的破口；且送的是「整段」而非「該字所在句」，
  外送量遠超需求（隱私章節 §9 的精神）。
- **建議**：在 `gemini_define` 進入點先 `filter::classify(&sentence)`（或至少 `looks_like_secret`），命中 `Secret`
  就回 `DEF_ERROR_EVENT`「已略過可能的機密內容」、不送出；並考慮只送「該字所在句」而非整個 textarea。

## 🟡 中（功能缺陷／錯誤處理）

### M1. event tap 被系統停用後只記 log、不重啟 → 雙擊 ⌘C 靜默失效
- **位置**：`src-tauri/src/monitor/mod.rs:84-86`（`TapDisabledByTimeout | TapDisabledByUserInput => log::warn!(...)`，註解自承「重新啟用列為待辦」）。
- **問題**：callback 太慢或使用者操作會讓系統停用 tap。目前**沒有重新 enable，也沒通知前端**，雙擊就此死掉直到重啟 App。
- **為什麼是問題**：PRD §7.3 要求「監聽層異常時可優雅降級」。現況是「無聲死亡」，使用者只會覺得「突然不靈了」。
- **建議**：收到 disabled 事件時呼叫 `CGEventTap` 的 re-enable（保留 tap 參考），或重建 tap；無法恢復時用 tray/通知提示「雙擊已停用，請重啟」。

### M2. `workbench_translate` 沒有請求序號／取消 → 連續編輯可能顯示舊譯文
- **位置**：`src-tauri/src/workbench.rs:155-200`（`run_mt`）對照 `pipeline.rs:190-199`（Glance 有 `request_seq` 丟棄過期結果）。
- **問題**：原文框 debounce 400ms（`Workbench.tsx:20,179`）後送翻，但**每次都是獨立 await**。網路慢時兩筆 in-flight，
  較早送出的若較晚回，會用**過時譯文**覆蓋畫面。
- **建議**：比照 Glance 用 `AtomicU64` 序號，回來時若非最新就丟棄；或前端記錄 latest text、回填前比對。

### M3. Gemini 串流 client 只有 connect timeout，無讀取／idle timeout → 串到一半卡住會無限等
- **位置**：`src-tauri/src/workbench.rs:51-54`（`llm_client` 只設 `connect_timeout`，刻意不設整體 timeout 以免切斷串流）。
- **問題**：沒有整體 timeout 是對的（避免腰斬長回應），但**也沒有任何 idle/read timeout**。連線在首 token 後 stall，
  AI 字義會永遠停在「串流中」，不 done 也不 error。
- **建議**：加一個「兩次 chunk 之間」的 idle timeout（例如在 `stream_generate` 的 `chunk().await` 外包一層 timeout，
  逾時視為 `Network` 錯誤），保留整體不限時但避免無限掛住。

### M4. 多行內容夾帶機密（如貼整份 `.env`／設定）會整段送 MT
- **位置**：`src-tauri/src/monitor/filter.rs:48-72`（`looks_like_secret`：多行只殺 PEM/PGP 區塊，`KEY=value` 賦值規則
  在 `s.lines().count() > 1` 時 `return false`）。
- **問題**：`api_key=sk-...` 若混在多行 log/設定裡，整段被判為 `Text` → 含密碼那行一起送 Google/DeepL。
- **為什麼是問題**：這是 D3「log／程式碼照翻、寧可放行」的**已知取捨**，不是 bug；但它與紅線「疑似機密不送外部」
  在多行情境下其實有張力，值得在這次盤點重新確認可接受。
- **建議**（若要收緊）：多行時仍逐行掃 `secret_assignment_value` 與已知前綴 token，命中就遮該行再送（而非整段放行或整段殺）。

### M5. 測試缺口：pipeline 快取與前端純邏輯零覆蓋
- **位置**：`pipeline.rs` 的 `cache_get/cache_put`（LRU、CACHE_CAP 邊界）、`cache_key`、`routing_signature` 無單元測試；
  前端 `wordAtCaret`（`Workbench.tsx:312`）、def-seq 過濾、debounce 完全沒有測試框架。
- **問題**：filter／double_press／router／dictionary 邏輯都測得不錯，但 LRU 淘汰、快取鍵碰撞、`wordAtCaret`
  的邊界（標點、連字號、選取 vs 點擊）這些「容易默默錯」的點沒守住。
- **建議**：替 `cache_*` 與 `routing_signature` 補 Rust 單測；`wordAtCaret` 抽成純函式並加前端測試（Vitest）。

## 🟢 低（一致性／可維護性）

### L1. CSP 關閉（`security.csp: null`）
- `src-tauri/tauri.conf.json`。目前安全（只載本地 `index.html`、React 預設 escape、全 codebase 無 `dangerouslySetInnerHTML`/`innerHTML`/`eval`），
  但少了縱深防禦。建議設一個最小 CSP（`default-src 'self'`），未來若引入 markdown/HTML 渲染才不會踩雷。

### L2. 過濾層的「寧可錯殺」誤判範例
- `filter.rs:107-130`：純 40 碼 SHA-1（`is_long_hex`）、含連字號的 UUID（`is_password_like`，含符號+數字+字母）會被判 `Secret` 而**靜默不翻**。
  符合 D3 精神，但建議在 docs/issues 補記這些已知誤殺樣本，避免日後被當新 bug 重查。

### L3. `change_count()` 在 event-tap 背景執行緒讀 NSPasteboard
- `monitor/pasteboard.rs:18-20` + `monitor/mod.rs:128`。已註明「clipboard manager 慣例可背景讀」，但 NSPasteboard 非保證 thread-safe。
  低風險、可接受；列此供日後若出現詭異 changeCount 時的排查起點。

### L4. 文件／UI 與現況脫節
- `docs/PRD.md` 標題仍為「ClipTranslate AI」、§4.7 仍寫字典走「有道／Cambridge API／mdict」——已被 D7（ECDICT 本地）取代。
- `src/components/SettingsView.tsx:316` 提示「字典查詢免 key（**公開字典 API**）」——字典已是本地 ECDICT，非公開 API。
- 建議同步成現況（產品名 Sumi、字典＝本地 ECDICT），避免新進者被舊敘述誤導。

### L5. 依賴現況（無可疑項）
- `reqwest 0.13.4`（已移 `system-proxy`，見 issues #1）、`keyring 3.6.3`（v3，apple-native）、`rusqlite 0.40.1`（bundled）、
  `objc2*`、`core-graphics/foundation`——皆與決策一致；`rdev` 已正確移除（spike-01）。無新增可疑套件。

## 總評

整體資安基礎扎實：key 寫入只進 Keychain、前端拿不回明文（只有 `*_set` 布林）、無 `console.log`/`println!` 殘留、
log 普遍只記字元數與耗時、機密過濾規則設計用心且有完整單測、git 歷史乾淨、`.gitignore` 到位。
真正要擔心的是**兩條把資料／金鑰送出邊界的破口**——它們都不是「設計錯」，而是「某條 edge 路徑沒套到既有防線」。

**最該優先處理的前 3 件：**
1. **H1 — Google key 進 log 的破口**：紅線等級。最小修法是 log 端對 `Network` 錯誤 redact，根治是把 Google key 改放 header。
2. **H2 — 字典 fallback 繞過機密過濾**：紅線等級。在 `gemini_define` 入口補 `filter::classify`，並只送該字所在句而非整段。
3. **M1 — event tap 停用後無聲失效**：這是「使用者體感產品壞掉」最常見的來源，補上 re-enable + 降級提示，對穩定性回報價值最高。
