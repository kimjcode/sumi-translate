# 決策筆記 — 翻譯來源與成本策略

**狀態：** P0 已選定其一，其餘刻意延後
**問題：** 使用者怎麼取得翻譯服務？流量與費用誰來扛？

這份筆記記錄選項與取捨，避免之後（或 Claude Code）從零重新爭論。

---

## 四條路與取捨

| 路線 | 怎麼運作 | 優點 | 代價 |
|---|---|---|---|
| **A. 自帶 key + 官方付費 API**（P0 選這條） | 使用者各自申請 Google/DeepL 的 key，存進自己的 Keychain | 穩定、合規、隱私可控、成本落在使用者身上（開發者不付錢） | 第一次有 onboarding 門檻（要去申請 key） |
| **B. 非官方網頁端點** | 模仿瀏覽器去打 Google/DeepL **網頁版**的內部 API，不用 key | 完全免費、開箱即翻、零門檻 | 脆弱（對方一改端點/加驗證就壞）、踩 ToS 灰色地帶、不適合商業化。Easydict / Pot 的免費引擎多屬此類 |
| **C. 開發者吃成本 + 變現** | 開發者付官方 API 錢，靠訂閱/買斷回收（如 Bob） | 使用者零門檻 | 開發者要付費、需要商業模式與金流 |
| **D. 本地模型** | 翻譯模型打包進 app，在使用者機器上跑 | 零 API 成本、隱私最佳、離線可用 | app 變肥、品質與速度受本機效能限制 |

---

## 各路適合的目標

- 要**完全免費 + 開箱即用** → B（接受「會壞」的脆弱）。
- 要**穩定 + 合規 + 隱私** → A（接受 onboarding 門檻）。
- 要**商業化** → C。
- 要**離線 / 極致隱私** → D（之後可作為 A 的隱私升級選項，呼應 PRD §9）。

---

## 目前決策（P0）

走 **A：自帶 key + 官方付費 API**。

- 預設 provider：**Google Cloud Translation**（付費 API 不拿內容訓練，與 §9 隱私賣點一致）。
- 選用：**DeepL**（品質較好，但免費層可能訓練，切換時當下提醒）。
- key 一律存 macOS Keychain。

**理由：** A 是唯一同時滿足「穩定 + 合規 + 隱私」的路，而隱私正是 Sumi 對工程師的核心賣點。onboarding 門檻是已知代價，用 §8 的引導流程把它做順即可。

---

## 開源硬限制（不可違反）

Sumi 開源、BYO-key——**絕不可把開發者自己的 key 打包進釋出版**。一公開等於全世界共用你的額度、刷爆你的帳單。每個使用者用自己的 key。開發者的 key 只用於本機開發測試。

---

## 之後要重新評估的點（現在不做）

核心未決問題：**要不要為了降低門檻，多接一個免 key 的免費引擎當預設？**

- 可能做法：把 B 當「開箱即用的入門引擎」，A 當「進階/隱私引擎」，使用者想要穩定與隱私再去設 key。
- 但 B 的脆弱性與 ToS 風險要先想清楚，且與「隱私賣點」可能衝突（網頁端點同樣把內容送出去）。

**重新評估的觸發條件：** Glance（P0）跑順、且開始有真實使用者回報「申請 key 太麻煩」之後，再回來做這個決策。現在不分心。

---

## 傾向（非定案）

P0 先把 A 做到能動、能日用。等門檻真的成為使用者流失的痛點，再認真評估「A + 一個更慷慨的免費引擎當預設」的混合方案——而不是一開始就為了零門檻去賭 B 的脆弱性。
