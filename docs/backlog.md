## 拼字 / 錯字辨識

**狀態：** 延後（Workbench 核心穩定後再評估）

**撿起來時別忘的原則：**
- **標示、不擅改**：翻使用者看到的原文，錯字另外標出來讓使用者自己決定。不要 silently 改了再翻——會改變語意，也讓人搞不清楚到底翻的是哪一句。
- **別動程式碼與識別字**：Sumi 的原文常是複製來的 log、變數名、snake_case，那些「錯字」是故意的。錯字辨識不能太兇，跟過濾層「別過度干涉工程師內容」是同一條原則。

**兩條低成本、之後可順手做的點子：**
1. **可編輯原文欄開原生 spellcheck**（WKWebView 的 spellcheck 屬性）→ 系統直接給紅波浪線 + 右鍵建議正確字。幾乎零成本，最快的落點。
2. **字典查無 → did-you-mean**：點選的字字典查不到（多半是拼錯/變形）時，給「你是不是要查 X？」。用字典 API 自帶建議，或用已接好的 Gemini 補一句。適合當 Workbench 核心做完後的小尾巴。

**是真功能、需獨立任務（別塞進現有任務）：** 打字即時 autocorrect、inline 建議 UI、整段 proofread。屬 P2 等級。

## Glance 長文限高

**狀態：** 觀察中（先用目前無限高的版本，累積體感再決定）

**問題：** Glance 譯文很長時會一直長高，與「快閃一瞥」的角色不符，也讓定位的「翻轉判斷」較難算準（高度不固定）。

**待決選項：**
- A. max-height ~240px + 窗內捲動。
- B. 超過就顯示「⌘↩ 展開看全文」導向 Workbench（較貼合 Glance 角色，偏好此案）。

**順手可消的隱患：** 高度固定後，「翻轉判斷用到上一次視窗高度」的問題自然消失。

**重新評估觸發：** 實際用下來發現長譯文在 Glance 裡夠煩的時候再做。

## Glance 單字分流走 ECDICT 字典

**狀態：** 延後（等 Workbench 的 ECDICT 字典層先落地）

**想法：** Glance 收到內容時分流——單一個字 → 查 ECDICT 顯示字典詞條（音標/詞性/中文）；一句/一段 → 照舊走 MT 翻譯。

**為什麼值得做：** 呼應做這工具的初衷——「忘記某個字的中文/英文，⌘CC 一下就知道」。MT 翻單字常只給生硬對應、沒音標詞性；ECDICT 資訊更豐富、離線零延遲。

**判斷規則（初擬）：** 去掉前後空白後，沒有空格／只有一個詞 → 走字典；否則走翻譯。多字詞（New York）再決定。

**為什麼排在 Workbench 之後：** Glance 與 Workbench 共用同一個 ECDICT 查詢層。先在 Workbench 把那層做穩，Glance 幾乎免費接上。

**重新評估觸發：** Workbench 的 ECDICT 字典落地、查詢層穩定之後。

## 整段語法糾正

**狀態：** 延後（P2；ECDICT 純字典版收乾淨、用一陣子後再規劃）

**想法：** 針對「選取的整段文字／整個原文」做語法糾正，**不是逐字**。掛在 Workbench 的選取動作或工具列「檢查語法」鈕，**不掛在點字上**（點字＝字典就夠）。

**為什麼從字典卡拿出來：** 點單字是「字」層級、語法糾正是「句/段」層級，是不同需求；當初綁在同一張點字卡是設計錯位。拆開後字典卡單純、這個功能也能好好設計。

**適用情境注意（重要）：** Sumi 的原文常是複製來的 log、程式碼、技術文件，不是使用者自己寫的句子。語法糾正只對「使用者自己寫、要校對的文字」有意義——別無差別對所有原文套用。

**待想清楚：** 觸發方式（應為使用者主動觸發，不自動跑）、呈現方式（指出哪裡有問題、建議怎麼改、能否套用）、要不要限定在使用者編輯過的原文才提供。

**重新評估觸發：** ECDICT 純字典版合併、用一陣子之後。


## 整段語法糾正 / 改寫（句、段級）

**狀態：** 延後（從字典卡移出，2026-06-16）

原本字典卡下段的「逐字 Gemini 文法/語境」已移除——逐字解釋價值低、又造成查無時雙請求 503。真正有價值的是**句／段級的語法糾正與改寫**（grammar check / rewrite），那是與「點單字查字典」不同層級的需求。

**撿起來時的方向：**
- 觸發點不是「點單字」，而是針對選取的句子／整段原文。
- 一個請求、串流回改寫建議，明確標示 AI。
- 與 Workbench 的「編輯原文→重翻」流程整合（改寫後可一鍵替換）。
- 屬獨立任務，先進 PRD 或另開任務再做。
