# 產品需求文件 (PRD) — V2

**產品名稱：** Sumi（墨）
**平台：** macOS（原生工具，第一階段不做 Windows / Linux）
**版本：** v2（修正內部矛盾、補回核心差異化、加入權限與隱私章節）
**最後更新：** 2026-06

---

## 0. 這版改了什麼（給自己看的修訂註記）

相較 v1，本版做了以下修正：

1. **救回兩個被弄丟的靈魂功能**：原文可編輯 + 即時重翻、逐字查字典/文法。v1 把產品收斂成唯讀翻譯彈窗，等於做一個功能更少的 Easydict，本版改回「雙模式」。
2. **解掉三組矛盾**：剪貼簿監聽 vs 雙擊觸發、不搶 focus vs 可編輯、<800ms vs 全走 LLM（見 §3 雙模式、§6 服務分層、§7 效能）。
3. **新增權限與 Onboarding 章節**（§8）：macOS 全域監聽必須給「輔助使用」權限，無法宣稱「0 設定」。
4. **隱私升級為差異化賣點**（§9）：排除規則、密碼類內容跳過、未來本地模型。
5. **修正快捷鍵**：全文 `Ctrl+C` → `⌘ Cmd+C`（macOS 複製鍵）。
6. **補上 Gemini** 並設為預設 LLM provider。
7. **新增中英混排處理規則**與可設定目標語言。

## 1. 產品定位

### 1.1 一句話定義
一個 macOS 常駐工具：複製文字後一鍵跳出翻譯，能快速一瞥，也能就地展開成可編輯的雙語工作台，逐字查字典與文法。

### 1.2 產品靈魂
> **Copy anything → instantly understand it → and reshape it if you want.**
> 從「看懂」延伸到「改對」，這是與市面快查彈窗的根本差異。

### 1.3 與既有開源工具的差異化
市面成熟工具（Easydict、Pot）都是「唯讀快查」。本產品的護城河在 **Workbench 模式**：在浮窗內繼續編輯原文、即時重翻、逐字查字典/文法——把「翻譯彈窗」變成「邊讀邊改的雙語編輯/校對工作台」。Glance 模式負責養成每日習慣，Workbench 模式負責留存與差異化。

---

## 2. 目標使用者

**主要使用者（第一階段）：** 工程師（software / firmware / test engineer），需要閱讀英文技術文件、log、email，處於中英混合工作環境。

**次要使用者：** 學生（閱讀英文資料）、商務人士（email / report 校對）。

**設計上的關鍵推論：** 主要使用者會複製 log、Jira 內部描述、甚至誤複製到密碼——**隱私風險真實存在**，這同時也是對工程師最有說服力的賣點（見 §9）。

---

## 3. 核心模型：兩種模式，一個視窗

這是整份 PRD 的核心設計，所有矛盾都由此解決。

### 3.1 Glance 模式（預設 · P0）
| 屬性 | 設定 |
|---|---|
| 觸發 | 雙擊 `⌘ Cmd+C`（300ms 內，可調整） |
| Focus | **不搶 focus**（macOS non-activating panel） |
| 內容 | 串流顯示翻譯，唯讀 |
| 行為 | 跟隨游標出現；失焦或閒置數秒自動淡出；`Esc` 關閉 |
| 用途 | 90% 情境：快速瞄一眼意思，不打斷工作流 |

### 3.2 Workbench 模式（差異化 · P1）
| 屬性 | 設定 |
|---|---|
| 進入方式 | 在 Glance 浮窗按「展開」鈕或快捷鍵（如 `⌘↩`） |
| Focus | **拿取 focus**，視窗放大 |
| 內容 | 原文框**可編輯** → 改了就**即時重翻**（debounce）；點選單字 → 字典/文法卡片 |
| 用途 | 10% 情境：要改原文、查詞、確認文法、潤稿 |

### 3.3 為什麼這樣解掉所有矛盾
- **監聽 vs 雙擊**：雙擊為預設；always-on 剪貼簿監聽降為**選用模式**（§4.1）。
- **不搶 focus vs 可編輯**：拆成兩個模式，各自有清楚的 focus 行為。
- **<800ms vs LLM**：Glance 走「快翻 + 串流」（§7）；LLM 的深度理解只在 Workbench 按需觸發。
- **共用後端**：兩模式共用同一條資料管線（§6），Workbench 只是多一個會 focus、可編輯的 view，因此**第一天就設計進架構，總工時低於事後retrofit**。

## 4. 功能範圍

### P0（V1 必做 — Glance 模式可獨立上線）

**4.1 觸發機制**
- **預設**：雙擊 `⌘ Cmd+C`。第一次 Cmd+C 本就把文字放進剪貼簿，第二次（時間窗內）才叫出浮窗讀剪貼簿——不與正常複製衝突。
- **可調整**：雙擊判定毫秒數、觸發鍵皆為設定項。
- **備用觸發**：`⌘⇧T`（手動叫出 / 翻譯剪貼簿現有內容）。
- **選用模式**：always-on 剪貼簿監聽（複製即翻），預設關閉，給重度使用者開。

**4.2 剪貼簿讀取與過濾**
- 觸發時讀取剪貼簿純文字內容。
- 排除：圖片、空值、與上次相同的內容。
- **密碼/機密內容偵測**：疑似密碼、token、key 的內容**不送出、不翻譯**（見 §9）。

**4.3 語言判斷與目標語言**
- 自動偵測來源語言：EN / ZH / JP / KR。
- 預設目標語言：繁體中文（台灣）。
- **可設定固定目標語言**（避免自動判斷出錯）。
- **中英混排規則**：偵測「主要語言」決定翻譯方向；若混排比例接近，依使用者設定的固定目標語言處理。

**4.4 翻譯引擎（快翻路徑）**
- 預設走**快速 MT**（Google Translate / DeepL API）取得即時首譯，搭配串流顯示，滿足 Glance 的速度需求。
- 結果可 cache（短句、重複內容）。

**4.5 Glance 浮窗 UI**
- macOS non-activating floating panel，always-on-top，不搶 focus。
- 結構：`原文 / 譯文（串流）/ [展開到 Workbench] 鈕`。
- `Esc` 或失焦關閉。

### P1（差異化 — Workbench 模式）

**4.6 可編輯工作台**
- 原文框可編輯；編輯後 debounce 觸發即時重翻。
- 譯文可直接複製。

**4.7 逐字字典 / 文法（關鍵：字典 ≠ LLM）**
- **字典查詢**（音標、詞性、中文釋義）→ 接**本地 ECDICT 英漢 SQLite**（離線、零外送、不編造），打包進 app；詳見 [decisions.md D7](decisions.md)。快、準、不依賴網路。
- **字典查無時的 AI 字義** → 才交給 **LLM（Gemini）**，按需觸發單一請求（明確標示「AI 推測」）。
- 互動：點選單字 → 跳字典卡片（查無才 fallback 到 Gemini）。逐句文法已收斂，見 D7 與 backlog。

**4.8 LLM provider 抽象層**
- 可切換 provider：**Gemini（預設）**、OpenAI、Claude、Google MT（fallback）。
- provider 以介面抽象，方便日後新增（含未來本地模型）。

**4.9 翻譯 Prompt（範例）**
```
You are a professional translation assistant.

Rules:
- Preserve meaning, not literal translation
- Use natural Traditional Chinese (Taiwan)
- If technical context exists, use engineering terminology
- Avoid overly formal tone unless necessary
- Handle mixed Chinese/English input gracefully

Return:
1. Translation
2. Key terms (if any)
```

### P2（Backlog · V2+）
- 工程模式：log 解讀、bug report 生成、Jira issue 草稿。
- OCR 截圖翻譯、PDF overlay。
- 個人化：使用者字典、公司術語庫、偏好風格（formal / engineering）。
- Smart actions：偵測意圖（translate / summarize / explain / rewrite）。
- TTS 發音、生字匯出 Anki。

## 6. 技術架構（macOS 為主）

### 6.1 技術選型
- **框架：Tauri（Rust 後端 + Web 前端）**。理由：沿用既有 React 能力、體積約為 Electron 的 1/10、適合常駐；且 Pot 已用 Tauri 做到同類功能，有可參考的實作。
- **前端 UI：React**。
- **macOS only**，先不處理跨平台抽象。

### 6.2 分層架構（Glance / Workbench 共用）
```
┌─────────────────────────────────────────────┐
│  [1] 全域監聽層 (Rust)                         │
│      雙擊 Cmd+C 偵測 · 剪貼簿讀取 · 去重/過濾    │
│      (rdev / CGEventTap 等級，需輔助使用權限)   │
└───────────────┬─────────────────────────────┘
                ↓
┌─────────────────────────────────────────────┐
│  [2] 語言/路由層                               │
│      語言偵測 · 目標語言 · 密碼內容跳過          │
└───────────────┬─────────────────────────────┘
                ↓
┌─────────────────────────────────────────────┐
│  [3] 服務抽象層 (核心設計)                      │
│   ├─ MT Provider     (Google/DeepL) ← 快翻     │
│   ├─ LLM Provider    (Gemini 預設)  ← 理解/文法 │
│   ├─ Dictionary      (本地 ECDICT)  ← 真字典    │
│   └─ Cache                                     │
└───────────────┬─────────────────────────────┘
                ↓
┌─────────────────────────────────────────────┐
│  [4] 視窗管理層 (Tauri/Rust)                   │
│   ├─ Glance: non-activating panel (不搶focus)  │
│   └─ Workbench: 正常視窗 (拿focus, 可編輯)      │
└───────────────┬─────────────────────────────┘
                ↓
┌─────────────────────────────────────────────┐
│  [5] UI 層 (React)                            │
│      原文(可編輯) · 譯文(串流) · 字典/文法卡片   │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  設定/狀態層: API key · 快捷鍵 · 偏好 (本地檔案) │
└─────────────────────────────────────────────┘
```

### 6.3 macOS 特定技術點
- **雙擊偵測**：需全域鍵盤事件監聽（CGEventTap / rdev 等級），無法只靠 Tauri 內建 global-shortcut（它註冊的是單一組合鍵，偵測不了「雙擊」與時間窗）。
- **Glance 浮窗**：使用 non-activating panel（不搶 focus、不奪走前景 App 的鍵盤焦點），對應 macOS NSPanel 的 nonactivating 行為。
- **剪貼簿**：Tauri clipboard plugin 或 Rust `arboard` crate。
- **串流**：LLM provider 的 streaming 輸出逐 token 渲染，體感「秒回」。

## 7. 非功能需求

### 7.1 效能（解掉 v1 矛盾）
- **Glance 體感 < 800ms**：靠「快速 MT + 串流」達成，**不**依賴 LLM 冷啟動。
- **Workbench 的 LLM 深度回應**：可接受 1–3s，因為是使用者主動展開後的進階操作。

### 7.2 成本控制
- 短句 / 重複 → cache。
- 快翻 → MT（便宜）。
- 深度理解 / 文法 → LLM，按需觸發（不是每次都呼叫）。

### 7.3 穩定性
- 不影響系統原本的複製行為。
- 不 crash 系統 input flow。
- 監聽層異常時可優雅降級（停用雙擊，仍可手動觸發）。

## 8. 權限與 Onboarding（v1 漏掉、必補）

macOS 全域鍵盤監聽**必須授予「輔助使用 (Accessibility)」權限**，使用者第一次開一定會被系統擋。因此「0 設定就能用」**不成立**，必須把它設計成順暢的引導步驟：

1. 首次啟動 → 友善說明「為什麼需要輔助使用權限」（不存取你打的字，只偵測觸發鍵）。
2. 一鍵帶使用者到「系統設定 → 隱私權與安全性 → 輔助使用」。
3. 偵測權限狀態，授權完成後自動進入可用狀態。
4. 第二步：引導填入 API key（Gemini）或選擇免 key 的快翻引擎。

**設計原則修正**：v1 寫的「0 learning curve / 不需設定就能用」改為「**權限與 key 設定一次，之後 0 操作**」。

## 9. 隱私（升級為差異化賣點）

主要使用者是工程師，會複製 log、內部 Jira、可能誤複製密碼。每次翻譯都把這些丟給第三方 LLM API 是真實風險。因此隱私不只是合規，而是**對工程師最有說服力的賣點**。

- **不永久儲存** clipboard 內容（本地歷史為選用、預設關閉）。
- **密碼/機密偵測**：疑似密碼、API key、token 的內容自動跳過、不送出。
- **排除規則**：可設定特定 App 或樣式的內容不觸發 / 不送出。
- **引擎選擇透明化**：清楚標示哪些是本地、哪些送雲端。
- **未來路線**：支援本地模型（Ollama 等），讓敏感內容全程不出機器。

## 10. 成功指標 (KPI)

**使用 KPI**
- 每日使用 > 10 次。
- 使用後不再開 Google Translate 網頁。
- 形成肌肉記憶（複製 → 雙擊 → 翻譯）。

**模式分工 KPI**
- Glance：日活與每日觸發次數（衡量習慣養成）。
- Workbench：展開率與展開後的編輯/查詞次數（衡量差異化是否被使用）。

**UX KPI**
- Glance 體感 < 1 秒。
- 不打斷工作流。
- 權限/key 設定一次完成率（onboarding 是否順）。
