## 範圍一：資安稽核（最重要，優先做）

對照 `CLAUDE.md` 的紅線逐條查，並找出任何偏離：

- **API key / secret**：是否真的只存 macOS Keychain？有沒有任何路徑會把 key 寫進 log、檔案、前端、錯誤訊息、或回傳給渲染層？整個 git 歷史有沒有殘留過真實 key？
- **剪貼簿內容**：有沒有任何地方把剪貼簿/原文內容寫進 log、暫存檔、或非必要地送往外部？（紅線：log 只能記字元數與耗時，不含內容。）
- **機密內容過濾**：疑似密碼/token/key 的過濾在所有送外部 API 的路徑上都有生效嗎？（包含 Glance、Workbench 編輯重翻、語言配對的雙次 MT、字典 fallback 等所有出口）有沒有繞過過濾的漏網路徑？
- **外部請求**：所有對 Google MT / DeepL / Gemini / ECDICT 下載的請求，URL、參數、錯誤處理是否安全？錯誤訊息會不會洩漏內部資訊或 key？
- **權限**：Accessibility（CGEventTap）的使用範圍是否最小必要？有沒有監聽了不需要的東西？
- **依賴**：Cargo / npm 依賴有沒有已知有問題或可疑的？（不必跑掃描工具，先憑判斷點出可疑項）
- **.gitignore**：secret、字典大檔、build 產物是否確實都擋住？

## 範圍二：功能與健全性測試

- **單元測試覆蓋**：現有測試涵蓋了哪些、明顯缺了哪些關鍵邏輯（過濾規則、雙擊判定、語言路由、詞形還原、快取鍵）？哪裡的 edge case 沒測？
- **錯誤處理**：外部服務失敗（503/429/網路斷/key 失效）時，每條路徑都有優雅處理嗎？有沒有會 panic、卡死、或把原始錯誤丟給使用者的地方？
- **狀態 / 生命週期**：視窗開關（Glance 隱藏、Workbench、設定視窗關閉=隱藏）、session 快取清除，有沒有殘留、洩漏、或重複開啟的問題？
- **邊界輸入**：超長文字、空白、純符號、混排、非預期語言，各路徑會不會出錯？
- **並發 / 時序**：雙擊偵測、changeCount 比對、串流，有沒有競態風險？

## 範圍三：一致性與可維護性（次要）

- 程式碼有沒有跟 `decisions.md` 記載的決策矛盾的地方？
- 有沒有重複邏輯、該共用卻各寫一套的（例如查詢層、過濾層）？
- 有沒有 dead code、留下的 debug 痕跡？
